**Findings — all NEGATE or OOS:**

1. **CROSS-PROTOCOL ststx-ratio manipulation (the headline angle) — NEGATE, traced 3 ways + WATCHLIST fragility.** `get-ststx-ratio-v3 = (reserve.get-total-stx − ststxbtc-supply) × 1e6 / ststx-supply`, and **`reserve-v1.get-total-stx = (stx-get-balance reserve) + stx-stacking`** — i.e. it INCLUDES the reserve's RAW STX balance (line 47-53), a real deviation from the tracked-var ideal → the ratio IS in-tx inflatable by transferring STX to the reserve. **But non-exploitable:** (a) **unrecoverable in-tx** — every reserve STX-out (`request-stx-for-withdrawal`, `request-stx-to-stack`, `get-stx`) is `dao.check-is-protocol`-gated, so an external attacker cannot pull injected STX back → no flash-loan round-trip; (b) **dilutes proportionally** — a donation X raises the ratio for ALL stSTX holders; an attacker holding S stSTX captures only `S·X/ststx-supply < X` → net loss on donate→redeem; (c) **LTV<1 bounds the cross-protocol over-borrow** — extra Zest borrow `= C·δ·LTV` where `δ=X/total-stx`, profitable only if `C·LTV > total-stx` (attacker collateral·LTV > whole reserve) — impossible since `C ≤ total-stx` and egroup enforces `LTV<BPS`. Normal deposit/withdraw don't move the ratio (numerator & denominator grow proportionally; mint/burn stSTX). NEGATE. **WATCHLIST/defense-in-depth (flag to both protocols):** the raw-balance-in-an-oracle-consumed-rate is fragile — it opens IF StackingDAO ever adds a recoverable STX-out path OR a consumer prices zstSTX at LTV≥1 OR a weaker-bounded consumer appears. Prefer tracked-var totals for externally-consumed rates; cross-protocol consumers should sanity-bound external rates. `[INSPECTED]`

2. **Reward-timing sandwich (deposit-before / withdraw-after a reward credit) — NEGATE.** stSTX is value-accruing (ratio grows when PoX rewards credit into reserve total-stx without minting stSTX). But withdrawal is **cycle-delayed**: `init-withdraw` → `withdraw` requires `burn-block-height ≥ unlock-burn-height` (= next-cycle-start + inset). So capturing a reward requires holding through the cycle = legitimate staking, no single-block sandwich. `withdraw-idle` allows same-cycle exit but only of IDLE (un-stacked) STX which earned NOTHING, minus a 1% `withdraw-idle-fee` → round-trip LOSS. NEGATE.

3. **Unstake/withdrawal queue — NEGATE.** `init-withdraw` SNAPSHOTS `stx-amount = ststx-amount × ratio` at init time into the NFT entry + locks it (`reserve.lock-stx-for-withdrawal`) + transfers the stSTX to core (burned only at `withdraw`). `withdraw` pays the RECORDED amount (not re-computed) after unlock + burns the stSTX. During the lock window both numerator (locked STX still in reserve balance) and denominator (stSTX not yet burned) are frozen → ratio undistorted for others. Snapshot insulates withdrawers from ratio moves; no attacker-triggerable ratio DROP exists (total-stx only falls via protocol-gated STX-out) → no lock-in-before-loss exploit. NEGATE.
